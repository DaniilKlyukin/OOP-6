# 1. Вступление: Проблема «хрупкого кода»

---

## Почему проекты умирают?

**Проблема «Карточного домика»**
* На старте проекта: новая фича за 2 часа.
* Через полгода: та же фича за 3 дня + 5 багов в случайных местах.
* **Хрупкость (Fragility):** изменение в одном модуле ломает логику в другом, никак не связанном с ним.

**Главный парадокс разработки**
Проекты редко закрываются из-за медленных алгоритмов. Чаще они закрываются, потому что стоимость внесения изменений превышает прибыль от этих изменений.

---

## Технический долг и стоимость владения

**Что такое Технический долг?**
* Это «быстрые и грязные» решения, принятые сегодня, чтобы успеть к дедлайну.
* **Тело кредита:** сэкономленное время сейчас.
* **Проценты:** дополнительное время, которое вы тратите *каждый раз* при работе с этим кодом в будущем.

**График Velocity (Скорость разработки):**
1. **Начало:** Взрывной рост (High Velocity).
2. **Накопление долга:** Постепенное замедление.
3. **Стагнация:** Команда тратит 90% времени на фикс багов и 10% на новые фичи.

---

## SOLID: Переосмысление

*Принципы SOLID — это не правила вежливости, это инструменты борьбы с энтропией.*

**S: Single Responsibility (Единственная ответственность)**
* Класс должен иметь только одну **причину для изменения**.
* Если вы меняете класс `User` и когда меняется БД, и когда меняется дизайн профиля — это плохой знак.

**O: Open/Closed (Принцип открытости/закрытости)**
* Программные сущности открыты для расширения, но закрыты для модификации.
* Идеальная архитектура позволяет добавить новый способ оплаты или тип врага в игре, **не меняя ни одной строчки** уже протестированного кода.

---

## Главный враг — Жесткая связность (Tight Coupling)

**Что такое Tight Coupling?**
Это состояние, когда объект `A` знает об объекте `B` слишком много: как он устроен, какие у него внутренние поля, как он сохраняет данные.

**Аналогия из реального мира:**
* **Жесткая связность:** Лампочка, провода которой припаяны напрямую к электросети внутри стены. Чтобы заменить лампочку, нужно ломать стену.
* **Слабая связность (Loose Coupling):** Розетка и вилка. Любой прибор с вилкой (интерфейс) может получить ток, не зная, как устроена электростанция.

---

## Куда мы движемся?

Чтобы код не превратился в спагетти, мы должны перейти от мышления «Я просто пишу функции в классах» к мышлению **«Я проектирую систему связей»**.

**Цели лекции:**
1. Понять, почему наследование — это «серебряная пуля», которая часто попадает в ногу.
2. Научиться собирать объекты из компонентов (Composition).
3. Построить «защитный периметр» вокруг бизнес-логики (DDD).

# 2. Проблема «Банана, Гориллы и Джунглей»

---

## Цитата Джо Армстронга (создатель Erlang)

> «Проблема объектно-ориентированных языков заключается в том, что они тянут за собой всё неявное окружение. Вам нужен был всего лишь **банан**, но в итоге вы получаете **гориллу**, держащую этот банан, и **целые джунгли** в придачу».

**В чем суть проблемы?**
Когда вы наследуетесь от класса, вы не просто берете одну функцию. Вы подписываетесь на всё его состояние, все его зависимости и всё поведение его предков.

---

## Ловушка глубокого наследования

**Иерархия «Пирамида смерти»:**
`Object` → `Entity` → `Actor` → `Pawn` → `Character` → `Hero` → `Warrior`

**Почему 5+ уровней — это зло:**
1.  **Непрозрачность:** Чтобы понять, как работает метод `attack()`, вам нужно просмотреть 6 файлов вверх по иерархии.
2.  **Хрупкий базовый класс (Fragile Base Class):** Изменение одной строки в классе `Actor` (уровень 2) может непредсказуемо сломать логику `Warrior` (уровень 7).
3.  **Избыточность:** Класс-потомок получает методы, которые ему не нужны, но которые он обязан «нести» на себе.

---

## Tight Coupling (Жесткая связь)

Наследование — это **самая сильная форма связи** в ООП. 

*   Вы не можете заменить «родителя» во время выполнения программы.
*   Если родительский класс требует в конструкторе 5 зависимостей (база данных, логгер, конфиг), каждый потомок **обязан** знать об этих зависимостях и прокидывать их вверх.
*   **Результат:** Тестирование одного маленького класса `Warrior` требует поднятия всей инфраструктуры, нужной его «предкам».

---

## Принцип: Composition over Inheritance

**Наследование (Inheritance):** Отношение **«IS-A»** (является).
*   *«Маг является Персонажем».*
*   Проблема: Маг не может перестать быть персонажем или стать «немножко мечом».

**Композиция (Composition):** Отношение **«HAS-A»** (содержит).
*   *«Персонаж имеет СпособностьПередвижения и СпособностьАтаки».*
*   Объект собирается из независимых деталей (компонентов) как конструктор.

---

## Живой пример: Система персонажей в RPG

### Плохой подход: Иерархия (Inheritance)
*   `Character` (can move)
    *   `Warrior` (move + sword attack)
    *   `Mage` (move + magic attack)
*   **Проблема:** Появился `BattleMage`. От кого наследоваться? 
    *   От `Warrior`? Тогда он получит ненужные методы владения щитом.
    *   От `Mage`? Тогда придется копипастить код удара мечом из `Warrior`.
    *   **Итог:** Дублирование кода или «Ромбовидное наследование».

---

### Хороший подход: Композиция (Pattern Strategy)

Вместо того чтобы прописывать логику атаки внутри класса, мы «вкладываем» её в него.

```csharp
// Интерфейс поведения
interface IAttackStrategy {
    void Attack();
}

// Конкретные реализации
class SwordSwing : IAttackStrategy { ... }
class FireballCast : IAttackStrategy { ... }

class Character {
    private IAttackStrategy _attackStrategy; // Композиция

    public void SetAttackStrategy(IAttackStrategy strategy) {
        _attackStrategy = strategy;
    }

    public void PerformAttack() {
        _attackStrategy.Attack();
    }
}
```

**Почему это лучше?**
1.  **Гибкость:** `BattleMage` — это просто `Character`, которому в рантайме можно подменить `SwordSwing` на `FireballCast`.
2.  **Тестируемость:** Можно протестировать `FireballCast` отдельно от класса `Character`.
3.  **Чистота:** Мы избавились от джунглей. Нам нужен банан (атака) — мы берем банан.

---

## Когда все же использовать наследование?

Наследование не запрещено, но оно должно быть:
1.  **Неглубоким** (1-2 уровня).
2.  **Логическим**: когда вы на 100% уверены, что связь не изменится (например, `Circle` всегда будет `Shape`).
3.  **Абстрактным**: наследуемся от интерфейсов или чистых абстрактных классов, а не от «тяжелых» классов с логикой.

# 3. Интерфейсы как контракты, а не просто «заглушки»

---

## Программирование на уровне интерфейсов

**Классическая ошибка:** Мыслить классами-реализациями.
* *«Мне нужен класс `SqlDatabase`, чтобы сохранить юзера».* — **Это тупик.**

**Правильный подход:** Мыслить контрактами.
* *«Мне нужен кто-то, кто умеет сохранять юзеров (интерфейс `IUserRepository`), а как он это сделает — мне неважно».*

**Почему это спасает архитектуру:**
1. **Изоляция:** Бизнес-логика не знает о деталях реализации (SQL, API, файлы).
2. **Параллельная разработка:** Один разработчик пишет логику регистрации, другой — реализацию БД. Они договорились только об интерфейсе.
3. **Легкое тестирование:** Вы можете подсунуть «фейковый» интерфейс (Mock) вместо реальной базы данных.

---

## Инверсия управления (IoC) и Dependency Injection (DI)

**Проблема:** Если класс сам создает свои зависимости через `new`, он намертво к ним привязан.
```csharp
class OrderService {
    // ЖЕСТКАЯ СВЯЗЬ: Мы прикованы к SQL и конкретному логгеру
    private MySqlDatabase _db = new MySqlDatabase(); 
}
```

**Принцип DI: «Не создавай зависимости сам, проси их у системы»**
Объект не должен искать ресурсы. Он должен объявлять, что они ему нужны (обычно через конструктор).

**Аналогия:**
* **Без DI:** Вы идете в лес, рубите дерево, строите верстак, чтобы сделать себе кофе.
* **С DI:** Вы просто говорите: «Мне нужен кофе». И система (DI-контейнер) приносит его вам готовым.

---

## Пример: Инъекция зависимости через конструктор

```csharp
class OrderService {
    private readonly IRepository _repository;

    // Класс не знает, какую БД ему дадут. 
    // Он просто работает с контрактом IRepository.
    public OrderService(IRepository repository) {
        _repository = repository;
    }

    public void PlaceOrder(Order order) {
        _repository.Save(order);
    }
}
```

---

## Кейс: Замена SQL на NoSQL без боли

**Дано:** Наш стартап вырос. SQL-база перестала справляться с нагрузкой, нужно перейти на MongoDB (NoSQL).

**Сценарий А (Плохой код):**
* У нас 50 классов, где написано `new MySqlDatabase()`.
* Нам нужно переписать все 50 классов, изменить типы данных и заново всё протестировать.
* **Результат:** 2 недели работы и куча новых багов.

**Сценарий Б (Архитектура на интерфейсах):**
1. Создаем новый класс `MongoRepository`, который реализует уже существующий интерфейс `IRepository`.
2. В конфигурации проекта (DI-контейнере) меняем одну строчку:
   `services.AddScoped<IRepository, MySqlRepository>()` 
   → на → 
   `services.AddScoped<IRepository, MongoRepository>()`.
3. **Результат:** Бизнес-логика (класс `OrderService`) даже не «заметила» подмены. Всё работает на новых рельсах.

---

## Главный вывод

Интерфейс — это **граница** между вашей уникальной логикой (бизнесом) и внешним миром (инфраструктурой).

* **Бизнес-логика:** "Что мы делаем?" (Продать товар, рассчитать скидку).
* **Инфраструктура:** "Как мы это храним/передаем?" (SQL, MongoDB, JSON, HTTP).

**Золотое правило:** Инфраструктура должна зависеть от Бизнеса (реализовывать его интерфейсы), а не наоборот.

# 4. Введение в DDD: Код на языке бизнеса

---

## Что такое «Домен»?

**Домен (Предметная область)** — это та сфера реальности, для которой вы пишете программу (Банкинг, Логистика, Геймдев, Медицина).

**Главная проблема:** Программисты часто думают таблицами в базе данных, а бизнес-заказчики — процессами («отгрузка», «верификация», «лимит»). 
Из-за этого возникает «трудности перевода»: в голове у заказчика одно, в коде — другое.

**Domain-Driven Design (DDD)** — это подход, при котором структура кода полностью повторяет структуру бизнеса. 
* Мы не просто пишем классы, мы создаем **модель реальности**.

---

## Entity (Сущность)

**Entity** — это объект, который определяется своей **идентичностью (ID)**, а не набором полей.

* **Пример:** Пользователь (`User`).
* Если пользователь сменил фамилию, почту и пароль — это всё еще тот же самый пользователь. 
* У него есть уникальный ключ (ID), который неизменен на протяжении всей «жизни» объекта.

**Свойства Entity:**
1. Имеет уникальный идентификатор.
2. Изменчив (Mutable): поля могут меняться.
3. Сравнивается только по ID.

---

## Value Object (Объект-значение)

**Value Object** — это объект, который определяется **содержимым своих полей**. У него нет ID.

* **Пример:** Деньги (`Money`), Адрес (`Address`), Цвет (`Color`).
* Если у вас есть две купюры по 100 рублей — вам не важно, какая из них какая. Они взаимозаменяемы, так как их ценность (значение) одинакова.
* Если вы измените в объекте `Money` поле `Amount` с 100 на 200 — это уже **другие деньги**, а не те же самые.

**Свойства Value Object:**
1. Нет идентификатора.
2. **Неизменяемость (Immutability):** если нужно изменить значение, мы создаем новый объект, а не правим старый.
3. Сравнивается по значениям всех полей.

---

## Практическая польза неизменяемости (Immutability)

Почему `Value Object` должен быть неизменяемым?

```csharp
// Проблема:
var price = new Money(100, "USD");
order1.Price = price;
order2.Price = price;

// Кто-то где-то в коде решил применить скидку:
order1.Price.Amount = 50; 

// Итог: цена order2 тоже стала 50! Это баг «скрытого изменения».
```

**Решение через Immutability:**
Мы не можем написать `price.Amount = 50`. Мы обязаны создать `new Money(50, "USD")`. Это гарантирует, что данные не «поплывут» в непредсказуемых местах.

---

## Агрегаты (Aggregates)

В сложной системе объекты связаны друг с другом в огромные графы. Если разрешить менять любой объект в любое время, данные в базе превратятся в кашу.

**Агрегат** — это группа связанных объектов, которые рассматриваются как единое целое для обеспечения целостности данных.

* **Пример: Заказ (`Order`) и Позиции заказа (`OrderItem`).**
* Правило бизнеса: «Сумма заказа не может быть больше 100 000 руб».
* Если мы разрешим менять `OrderItem` напрямую (в обход `Order`), мы можем случайно нарушить это правило.

**Правила Агрегата:**
1. У агрегата есть **Корень (Aggregate Root)** — единственный вход в систему. В примере выше это `Order`.
2. Внешний мир может общаться только с Корнем. Нельзя залезть внутрь заказа и поменять цену позиции напрямую.
3. Агрегат гарантирует, что все бизнес-правила (инварианты) внутри него соблюдены.

---

## Итог: Зачем это нужно?

1. **Единый язык (Ubiquitous Language):** Программист и менеджер говорят на одном языке. Если в бизнесе есть понятие «Анкета», в коде должен быть класс `Questionnaire`, а не `UserDataTable`.
2. **Безопасность:** Value Objects и Агрегаты защищают данные от некорректных состояний.
3. **Читаемость:** Код начинает выглядеть как описание процесса, а не как перекладывание байтов.

# 5. Чистая архитектура (Clean Architecture) в миниатюре

---

## Архитектура «Луковицы» (Onion Architecture)

Чистая архитектура — это способ организации кода, при котором бизнес-логика не зависит от фреймворков, баз данных и внешних интерфейсов.

**Основные слои:**
1.  **Domain (Ядро):** Сущности, Value Objects, бизнес-правила (то, что мы обсуждали в DDD). Это сердце системы.
2.  **Application (Сервисы/Use Cases):** Описывает, *что* программа делает. Здесь находятся сценарии (например, «Зарегистрировать пользователя», «Оформить возврат»).
3.  **Infrastructure (Внешнее кольцо):** Реализация БД, API-клиенты, логирование, UI. Это детали реализации.

---

## Главное правило: Зависимости направлены внутрь

Это фундаментальный закон. Внутренние слои **ничего не знают** о внешних слоях.

*   **Домен** не знает, что он будет сохраняться в SQL или MongoDB.
*   **Сервисы** не знают, откуда пришел запрос — из браузера, мобильного приложения или консоли.
*   **Инфраструктура** знает обо всех, она лишь реализует контракты, которые определил Домен.

**Почему это важно?**
Если вы решите сменить библиотеку для отправки Email, вы измените только внешний слой. Код, отвечающий за логику «когда и кому отправить письмо», останется нетронутым.

---

## Где в этой схеме место ООП?

В Чистой архитектуре ООП — это не просто способ организации кода, это **инструмент защиты**.

1.  **Инкапсуляция как забор:** Мы скрываем внутреннее состояние Агрегата, чтобы никто из слоя «Инфраструктура» не мог напрямую поменять баланс счета, минуя бизнес-проверку.
2.  **Полиморфизм как механизм стыковки:** Интерфейсы определяются в центре (в Домене), а их реализация подсовывается снаружи (через DI). 
    *   *Пример:* Домен говорит: «Мне нужен `IMessageSender`». Инфраструктура дает `TelegramSender`. Домену всё равно, как работает Telegram API.

---

## Слои в коде (Наглядная схема)

*   **Infrastructure Layer:**
    *   `SqlUserRepository.cs`
    *   `SendGridEmailService.cs`
    *   `UsersController.cs` (Web API)
*   **Application Layer:**
    *   `RegisterUserUseCase.cs`
    *   `IUserRepository.cs` (Интерфейс!)
*   **Domain Layer:**
    *   `User.cs` (Entity)
    *   `Email.cs` (Value Object)

**Связь:** Контроллер вызывает UseCase. UseCase манипулирует Сущностями и сохраняет их через Интерфейс. Реальный SQL-репозиторий выполняет работу «в тени».

---

## Чистая архитектура vs «Спагетти»

| Признак | Спагетти-код | Чистая архитектура |
| :--- | :--- | :--- |
| **Связи** | Все знают обо всех | Зависимости строго в одну сторону |
| **Тестирование** | Нужна база данных и запущенный сервер | Можно тестировать логику без БД за миллисекунды |
| **Смена БД** | «Проще переписать проект» | «Замените один класс в Infrastructure» |
| **Порог входа** | Низкий (пишем как хотим) | Высокий (нужно проектировать) |

---

## Золотое правило архитектора

> **«Детали (БД, Фреймворки, UI) должны зависеть от Политики (Бизнес-логика). Политика никогда не должна зависеть от Деталей».**

Ваш код должен выглядеть так, будто он может работать в консоли так же легко, как в облачном сервисе. ООП здесь — ваш главный инструмент по созданию этих независимых границ.


# 6. Практический кейс: Рефакторинг

---

## Исходный код: «Всё-в-одном» (Spaghetti-style)

```csharp
// Проблема: Наследование ради доступа к БД, жесткая связь, каша из логики
public class OrderManager : SqlDatabase {
    public void ProcessOrder(string email, decimal amount, string currency, int type) {
        // 1. Валидация (Primitive Obsession)
        if (!email.Contains("@")) throw new Exception("Bad email");
        if (amount <= 0) throw new Exception("Bad amount");

        // 2. Бизнес-логика смешана с типами (Hardcoded logic)
        if (type == 1) { /* логика обычного заказа */ }
        else if (type == 2) { /* логика цифрового заказа */ }

        // 3. Прямая работа с инфраструктурой
        string sql = $"INSERT INTO Orders VALUES ('{email}', {amount})";
        this.ExecuteQuery(sql); 

        // 4. Прямая зависимость от внешнего сервиса
        var smtp = new SmtpClient("smtp.mail.com");
        smtp.Send("Order placed");
    }
}
```

**Что здесь плохо?**
*   **Inheritance:** `OrderManager` — это не база данных. Наследование здесь только ради метода `ExecuteQuery`.
*   **Primitive Obsession:** Почта и деньги — это просто строки и числа. Нет защиты от ошибок.
*   **Hardcoded Dependencies:** Невозможно протестировать метод, не отправляя реальное письмо и не имея базы.

## Шаг 1: Выделяем Value Objects

Избавляемся от «одержимости примитивами». Логика валидации переезжает внутрь самих объектов.

```csharp
public record Email(string Value) {
    public Email {
        if (!Value.Contains("@")) throw new ArgumentException("Invalid email");
    }
}

public record Money(decimal Amount, string Currency) {
    public Money {
        if (Amount <= 0) throw new ArgumentException("Amount must be positive");
    }
}
```

---

## Шаг 2: Композиция и Интерфейсы

Заменяем наследование от БД на внедрение зависимостей через интерфейсы.

```csharp
public interface IOrderRepository {
    void Save(Order order);
}

public interface INotificationService {
    void Notify(Email email, string message);
}
```

---

## Шаг 3: Итоговый «Чистый» код

Применяем паттерн **Strategy** для типов заказов и **DI** для инфраструктуры.

```csharp
public class OrderService {
    private readonly IOrderRepository _repository;
    private readonly INotificationService _notification;

    public OrderService(IOrderRepository repository, INotificationService notification) {
        _repository = repository;
        _notification = notification;
    }

    public void PlaceOrder(Email email, Money price, IOrderStrategy strategy) {
        // Создаем сущность (Entity), которая сама проверяет свои правила
        var order = new Order(email, price);
        
        // Применяем стратегию (Композиция вместо If/Else)
        strategy.Apply(order);

        // Сохраняем через абстракцию
        _repository.Save(order);

        // Уведомляем через абстракцию
        _notification.Notify(email, "Ваш заказ оформлен!");
    }
}
```

---

## Результат рефакторинга

1.  **Тестируемость:** Мы можем написать Unit-тест для `OrderService`, подставив `MockNotificationService`. Нам не нужен SMTP-сервер.
2.  **Гибкость:** Чтобы изменить способ сохранения (с SQL на JSON), мы просто создаем новый класс, реализующий `IOrderRepository`.
3.  **Читаемость:** Метод `PlaceOrder` теперь читается как рассказ, а не как инструкция к процессору.
4.  **Надежность:** Мы не можем создать объект `Email` с невалидными данными — система выдаст ошибку еще на входе.

## Главный вывод лекции

Многие считают, что ООП — это умение группировать функции в классы и строить длинные цепочки наследования. Это заблуждение.

**ООП — это искусство управления зависимостями.**

Хороший архитектор тратит 10% времени на написание логики и 90% времени на то, чтобы провести границы между модулями так, чтобы изменение в одном месте не вызывало пожар во всём проекте.

**Ваш чек-лист на ближайший проект:**
1.  **Stop Inheritance:** Прежде чем написать `extends` или `:`, спросите себя: «Могу ли я сделать это через интерфейс и поле в классе?».
2.  **Protect the Domain:** Не пускайте SQL-запросы или HTTP-детали в сердце вашей логики.
3.  **Think in Values:** Используйте Value Objects для всего, что не имеет уникального ID.
